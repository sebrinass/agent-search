## 一、搜索相关功能

### 1.1 混合检索 ✅ 已确认

**需求**：
- 对 SearXNG 返回结果重新排序，过滤不相关内容
- 使用 **RRF（Reciprocal Rank Fusion）** 融合 BM25 和语义嵌入

**为什么用 RRF**：
- BM25 分数和语义分数量纲不同，直接加权融合不公平
- RRF 只看排名，不看分数，天然解决量纲问题
- 无需调参，k=60 是业界标准

**配置**：
| 变量 | 默认值 | 说明 |
|------|--------|------|
| `ENABLE_EMBEDDING` | `true` | 设为 false 关闭 |
| `OLLAMA_HOST` | `http://localhost:11434` | Ollama 地址 |
| `EMBEDDING_MODEL` | `nomic-embed-text` | 嵌入模型 |
| `TOP_K` | `5` | 返回结果数量 |

**实现要点**：
- [x] 只支持 Ollama，删除 OpenAI 支持
- [x] 保留中文分词（Intl.Segmenter）
- [x] 删除文本分块逻辑（搜索结果很短，不需要）
- [x] 默认开启，需要手动关闭
- [x] 用 RRF 替代加权融合，无需调参

**新建文件**：`src/embedding.ts`（约 150 行）

---

### 1.2 链接去重 ✅ 已确认

**需求**：
- 同一次搜索内去重（SearXNG 多引擎可能返回相同链接）
- 后续搜索累计去重（TTL 时间内不重复返回已见过的链接）

**实现方式**：
- 全局共享一个去重池（无会话管理）
- TTL 自动过期，可配置
- 内存级别，服务重启后清空

**配置**：
| 变量 | 默认值 | 说明 |
|------|--------|------|
| `LINK_DEDUP_TTL` | `60` | 去重池过期时间（秒） |

**代码量**：约 30 行，独立模块

---

### 1.3 并发搜索 ✅ 已确认

**需求**：
- Research 工具的一部分
- 多个关键词并发执行搜索

**配置**：
| 变量 | 默认值 | 说明 |
|------|--------|------|
| `MAX_KEYWORDS` | `3` | 每次最多搜索几个关键词 |
| `SEARCH_TIMEOUT_MS` | `30000` | 单个搜索超时时间（毫秒） |

**实现要点**：

- 超时自动跳过

## 二、URL 读取相关功能

### 2.1 JS 渲染降级 ✅ 已确认

**需求**：fetch 失败时自动使用轻量 JS 渲染

**实现方案**：分层降级
```
第1层：fetch（获取原始 HTML）
    ↓ 失败或内容为空
第2层：Happy DOM（执行 JS 渲染，~1MB）
    ↓ 仍然失败
提示："请使用浏览器 MCP 处理"
```

**配置**：
| 变量 | 默认值 | 说明 |
|------|--------|------|
| `ENABLE_JS_RENDER` | `true` | 设为 false 关闭降级 |

**备选方案**：Playwright（完整浏览器，先不实现）
- 使用 ARIA Snapshot 获取干净内容
- 不需要 Readability（自带语义化输出）
- 比 Puppeteer 更适合（Puppeteer 不支持 ARIA Snapshot）

**依赖**：happy-dom（~1MB，必需依赖）

---

### 2.2 Readability 提取 ✅ 已确认

**需求**：提取页面正文，去除广告/导航等噪声

**实现方案**：
- 用于 Happy DOM 方案（轻量级）
- 提取正文内容，减少噪声

**配置**：
| 变量 | 默认值 | 说明 |
|------|--------|------|
| `ENABLE_READABILITY` | `true` | 设为 false 关闭提取 |

**依赖**：@mozilla/readability（~50KB，必需依赖）

**注意**：Playwright 备用方案使用 ARIA Snapshot，不需要 Readability

---

### 2.3 robots.txt 检查 ❌ 已删除

**原因**：
- 用户主动请求读取，类似浏览器访问
- 增加复杂度，实际作用小
- 网站拒绝时会返回 403 错误

---

### 2.4 阅读工具合并 ✅ 已确认

**需求**：合并单个/批量读取为统一工具

**新设计**：
```
read_content(urls) → 统一阅读
├── "https://a.com"                    → 读取1个
├── "https://a.com | https://b.com"    → 读取多个（并发）
└── ["https://a.com", "https://b.com"] → 数组形式
```

**优点**：
- 工具数量减少（2→1）
- 与搜索工具使用方式一致
- 底层统一并发处理

## 四、缓存系统

### 4.1 缓存策略 ✅ 已确认

**设计原则**：保持轻量，只保留必要的缓存

**缓存列表**：

| 序号 | 缓存 | 作用 | 大小 | TTL |
|------|------|------|------|-----|
| 1 | 链接去重池 | 避免返回重复链接 | 100 | 60秒 |
| 2 | URL 内容缓存 | 缓存页面内容 | 200 | 60秒 |
| 3 | 嵌入缓存 | 缓存嵌入向量（加速混合检索） | 500 | 30分钟 |

**嵌入缓存详细设计**：
- 依赖：`quick-lru`（仅 2KB）
- 存储格式：Float16Array（节省 50% 内存）
- 键：文本内容的 MD5 hash
- 不持久化（MCP 无状态服务）
- qwen3-embedding-0.6B 输出维度 1024，500 条约 1MB

**配置**：
| 变量 | 默认值 | 说明 |
|------|--------|------|
| `LINK_DEDUP_TTL` | `60` | 链接去重过期时间（秒） |
| `URL_CACHE_TTL` | `60` | URL 内容缓存过期时间（秒） |
| `URL_CACHE_SIZE` | `200` | URL 内容缓存最大条数 |
| `EMBEDDING_CACHE_SIZE` | `500` | 嵌入缓存最大条数 |

**删除的缓存**：
- 搜索结果缓存（SearXNG 自带）
- 语义缓存（复杂度高，收益低）
- 搜索去重（SearXNG 自带）

**删除的文件**：
- session-tracker.ts（业务追踪，与缓存重叠）
- robots.ts（用户主动请求，不需要检查）
- robots-txt-parser.d.ts（配套删除）

### 5.2 search 工具规范

**位置**：`src/research.ts` → `SEARCH_TOOL`

**描述**（精简版）：
```
增强搜索工具 - 规划思考 + 自动搜索

功能：
1. 记录结构化思考步骤
2. 自动执行 searchedKeywords 中的搜索（并发）
3. 返回思考状态 + 搜索结果

使用方法：
1. 填写 thought 描述当前思考
2. 填写 searchedKeywords 指定要搜索的词（最多5个，自动执行）
3. 根据返回的搜索结果决定下一步
4. 如需深读某个网页，调用 read 工具

返回内容：
- thoughtStatus: 思考状态（步骤编号、是否继续等）
- searchResults: 搜索结果（每个keyword的匹配结果，包含URL用于read）

注意：
- 每个关键词应是独立的实体，避免组合查询
- 搜索结果已按相关性排序
- 结果中的URL可用于后续 read 工具深入阅读
```

**必填参数**：
| 参数 | 类型 | 说明 |
|------|------|------|
| `thought` | string | 当前思考内容 |
| `thoughtNumber` | number | 当前思考步骤编号 |
| `totalThoughts` | number | 预计总思考步骤数 |
| `nextThoughtNeeded` | boolean | 是否需要继续思考 |

**可选参数**：
| 参数 | 类型 | 说明 |
|------|------|------|
| `searchedKeywords` | string[] | 要搜索的关键词列表（最多5个） |
| `site` | string | 限制搜索域名 |
| `informationSummary` | string | 上一步获取的关键发现 |
| `isRevision` | boolean | 是否修正之前的思考 |
| `revisesThought` | number | 修正哪个思考步骤 |
| `branchFromThought` | number | 从哪个思考步骤分支 |
| `branchId` | string | 分支标识 |
| `needsMoreThoughts` | boolean | 是否需要更多思考步骤 |

### 6.2 Context7 代码搜索集成 ✅ 已确认

**需求**：新增代码文档搜索能力

**架构**：
```
┌─────────────────┐         ┌─────────────────────────────────┐
│  本地 MCP       │  HTTP   │  Context7 官方 API              │
│  code_search    │ ──────► │  (api.context7.com)            │
│                 │         │                                  │
│  两个工具：      │         │  • resolve-library-id           │
│  code_resolve   │         │  • query-docs                   │
│  code_query     │         │                                  │
└─────────────────┘         └─────────────────────────────────┘
```

**工具命名**：方案 A（前缀统一 `code_`）
- `code_resolve`：解析库名为 ID
- `code_query`：查询库文档

**为什么集成**：
- Context7 爬取主流库的文档，提供实时更新的代码示例
- 解决 AI "幻觉"问题（生成过时的 API 代码）
- 无需本地数据库，调用官方 API 即可

**配置**：
| 变量 | 必填 | 说明 |
|------|------|------|
| `CONTEXT7_API_KEY` | 否 | 不填也能用，但有速率限制 |

**免费版限制**：
- 无 API Key：较低速率限制
- 有 API Key：更高速率限制（推荐申请免费 key）

**实现要点**：
- 直接调用 Context7 REST API，无需运行独立 MCP Server
- API 地址：`https://context7.com/api/v2/libs/search` 和 `/v2/context`
- 参考实现：d:\项目\context7-reference\packages\mcp\src\lib\api.ts

**依赖**：无需额外依赖（使用原生 fetch）
